# Лог архитектурных решений (ADR)

## ADR-001: Выбор Kafka для асинхронной коммуникации
- **Когда:** 2025-06-19
- **Контекст:** Нужно развязать сервисы, чтобы добавлять новые без изменения старых.
- **Решение:** Использовать Kafka как брокер событий.
- **Альтернативы:** RabbitMQ (более простой, но менее масштабируемый), Azure Service Bus (платный).
- **Последствия:** Повышение сложности на старте, но обеспечение гибкости в будущем.

## ADR-002: Каждый сервис имеет свою схему в PostgreSQL
- **Когда:** 2025-06-19
- **Контекст:** Микросервисы не должны иметь общих таблиц.
- **Решение:** Разные схемы для Identity, Gallery, Upload.
- **Альтернативы:** Общая БД (проще, но создаёт связанность).
- **Последствия:** Нет прямых JOIN между сервисами, нужно больше API-вызовов.
